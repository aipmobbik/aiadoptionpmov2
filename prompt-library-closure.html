Lessons Learned & Post-Project Review
Lessons Learned Report
Extract insights and best practices to improve future projects
Specific Heavy context ChatGPT · Claude · Gemini
▼

Input

  • Project Closure Report (scope, budget & timeline)
  • RAID Log (complete history, incl. resolved items)
  • Team Retrospective Survey (min. 5 responses)
💡 If unavailable, use status reports + meeting notes as a substitute for the closure report.

Output

  • Lessons Learned report (by People / Process / Technology)
  • Root cause analysis with recurrence risk rating
  • Top 5 recommendations for the next project

When to use

At project closure or end-of-phase review. Attach closure report + RAID + retrospective survey.

Template
You are a [role, e.g., Senior PMO Retrospective Facilitator]. Review the attached documents and generate a Lessons Learned report for [Project Name]. Attach: [project closure report + RAID log + retrospective survey responses]. Evidence-based only. [1-2] sentences per finding. [Director-level / Team-facing] tone. No speculation. Output: Executive Summary (2–3 sentences) What Went Well (by People / Process / Technology) Areas for Improvement (with root cause) Top 5 Recommendations Evaluation: Every finding must map to a source document. Flag any gap where evidence is insufficient.
Full example
You are a Senior PMO Retrospective Facilitator with 10+ years of post-project review experience. Generate a formal Lessons Learned Report for DM&A Phase 2 based exclusively on content found in the attached closure deck. Attach: The project closure report. Evidence-based only. 1–2 sentences per finding. Professional, Director-level tone. No speculation. Output: Executive Summary | What Went Well (People/Process/Technology) | Areas for Improvement | Top 5 Recommendations Evaluation: Every finding must cite a source. Flag any finding without supporting evidence.
Generated Output Example
LESSONS LEARNED — DM&A Phase 2 Project
Executive Summary
Phase 2 closed on all three formal KPIs: delivered on time (16 months, Meet Expectation), within 5% budget variance (Exceed Expectation), and with a customer satisfaction score of 8.48/10 (Very Good).
What Went Well
People: DG roles fully defined across 32 domains; Change Management activated newsletters, knowledge sessions, a Bootcamp, and four Learning+ courses.
Process: Phased “Think Big, Start Small, Grow Fast” structure; batched dashboard rollouts with controlled go-live; no-cost change request absorption.
Technology: Unified Data Lake + centralised Data Warehouse; standardised pipeline templates; ML model live 1 Sep 2023; consistent cross-BU dashboard metrics.
Areas for Improvement
Timeline extended from 12→16 months — root cause not documented.
ML model process integration incomplete — explicitly deferred to Phase 3.
Usage dipped during Nov–Dec 2023 issue resolution period.
No post-Phase 2 DG Maturity re-assessment score documented.
Top 5 Recommendations
1. Document root causes of timeline extension before Phase 3 baselining.
2. Prioritise end-to-end ML forecasting process integration as a Phase 3 day-one workstream.
3. Introduce a formal hypercare period between dashboard go-live and project phase close.
4. Conduct a post-Phase 2 DG Maturity re-assessment and establish a Phase 3 target state.
5. Formalise dashboard adoption tracking as an ongoing programme KPI.
